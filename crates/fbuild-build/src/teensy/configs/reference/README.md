# PlatformIO Reference Configs

These JSON files capture the **authoritative build flags** that PlatformIO uses
for each Teensy board: compiler flags, preprocessor defines, linker flags, and
linker libraries. They serve as the ground truth for validating that fbuild's
MCU configs and `BoardConfig::get_defines()` stay in sync with PlatformIO.

## How to regenerate

Run the extraction script (reads from `build/lib/fbuild/platform_configs/`):

```bash
# All platforms:
uv run python ci/extract_pio_build_flags.py --all

# Teensy only:
uv run python ci/extract_pio_build_flags.py --platform teensy

# Specific boards:
uv run python ci/extract_pio_build_flags.py --board teensy36 --board teensy41
```

## How tests use these

Rust tests in `mcu_config.rs` load both the fbuild MCU config and the
corresponding reference file via `include_str!`, then validate:

- **`test_compiler_flags_match_platformio_reference`** — every compiler flag
  from the reference is present in fbuild's MCU config (superset check).
- **`test_linker_flags_match_platformio_reference`** — every linker flag and
  library from the reference is present in fbuild's MCU config.
- **`test_defines_match_platformio_reference`** — every preprocessor define
  from the reference is generated by `BoardConfig::get_defines()`.

If a test fails, it means fbuild has drifted from PlatformIO. Either update
the MCU config / `board.rs` to include the missing flag, or regenerate the
reference if PlatformIO has changed its defaults.
